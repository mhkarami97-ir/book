## توضیحات
جامع‌ترین مرجع موجود درباره پروتکل HTTP که در سال ۲۰۰۲ توسط O'Reilly منتشر شد و نوشته دیوید گورلی و برایان توتی به همراه سه نویسنده دیگر است. کتاب در ۲۱ فصل و ۶۵۶ صفحه، نه‌تنها HTTP را توضیح می‌دهد بلکه تمام فناوری‌های هسته‌ای وابسته به آن — از مدیریت اتصال TCP، پراکسی و Cache، احراز هویت، HTTPS تا Redirect و Load Balancing — را با دیاگرام‌های دقیق پوشش می‌دهد. این کتاب بیشتر یک «چرا و چطور» است تا یک مرجع header-by-header.

## نظر
- `امتیاز` : 08/10
- `به دیگران توصیه می‌کنم` : بله
- `دوباره می‌خوانم` : بله
- `ایده برجسته` : مکانیزم **HTTP Connection Management** — کتاب به‌خوبی نشان می‌دهد که چرا HTTP/1.1 با Keep-Alive و Pipeline، و چرا HTTP/2 با Multiplexing، هر کدام تلاشی برای حل مشکل «Head-of-Line Blocking» بودند. این درک لایه‌ای از تکامل پروتکل، به مهندس کمک می‌کند که بهینه‌سازی performance را نه از روی حفظ بلکه از روی درک انجام دهد.
- `تاثیر در من` : وقتی روی API های .NET Core کار می‌کنم، حالا می‌دانم دقیقاً چه اتفاقی پشت هر Request می‌افتد — از TCP handshake تا Cache-Control header؛ این درک عمیق‌تر در Debug کردن مشکلات شبکه و طراحی درست Middleware ها تأثیر مستقیم داشته.
- `نکات مثبت` : کامل‌ترین توضیح HTTP/1.1 که در قالب یک کتاب یافت می‌شود؛ دیاگرام‌های فراوان که مفاهیم انتزاعی پروتکل را ملموس می‌کنند؛ پوشش عمیق موضوعاتی که اکثر کتاب‌های دیگر از آن‌ها عبور می‌کنند مثل Web Robots، Content Negotiation و Internationalization
- `نکات منفی` : HTTP/2 و HTTP/3 را پوشش نمی‌دهد چون قبل از آن‌ها نوشته شده؛ نمونه کدها به Perl و C نوشته شده‌اند که امروز بی‌ربط به نظر می‌رسند؛ برای کسی که فقط دنبال کار با API های مدرن است، عمق کتاب بیش از نیاز است

# فصل اول — بخش اول: HTTP چیست و چرا اهمیت دارد؟

## معرفی و جایگاه HTTP

نویسندگان کتاب با یک ادعای ساده اما قوی شروع می‌کنند: «HTTP زبان مشترک اینترنت مدرن است.»  این جمله را جدی بگیرید. وقتی می‌گویند «زبان مشترک»، منظورشان این است که تمام Web Browsers، Web Servers و Web Applications — بدون استثنا — برای تبادل اطلاعات روی HTTP تکیه می‌کنند. 

نکته‌ای که اینجا مهم است و بسیاری از توسعه‌دهندگان نادیده می‌گیرند: HTTP یک **Application-layer Protocol** است. این یعنی HTTP اصلاً نگران جزئیات شبکه نیست — آن جزئیات را به TCP/IP واگذار کرده.  این تقسیم مسئولیت (Separation of Concerns) در طراحی HTTP، یکی از تصمیمات معماری بنیادی این پروتکل است. 

## HTTP به‌عنوان یک Multimedia Courier

نویسندگان HTTP را «پیک چندرسانه‌ای اینترنت» می‌نامند.  چرا؟ چون HTTP روزانه میلیاردها فایل از انواع مختلف — تصویر JPEG، صفحه HTML، فایل متنی، فیلم MPEG، فایل صوتی WAV و Java Applet — را با سرعت، راحتی و قابلیت اطمینان جابه‌جا می‌کند. 

یک ویژگی حیاتی که کتاب روی آن تأکید می‌کند: HTTP از **Reliable Data-Transmission Protocols** استفاده می‌کند. این تضمین می‌دهد که داده شما در مسیر انتقال خراب یا تغییر نمی‌کند.  برای شما به‌عنوان یک توسعه‌دهنده .NET Core، این یعنی لایه HTTP زیرساخت انتقال را تضمین می‌کند و شما می‌توانید تمام انرژی خود را روی منطق اپلیکیشن متمرکز کنید. 

## Web Clients و Web Servers

کتاب ساختار اصلی را این‌گونه تعریف می‌کند: 

- **Web Server** (HTTP Server): داده‌های اینترنت را نگه می‌دارد و در پاسخ به درخواست‌ها ارائه می‌دهد
- **Web Client** (HTTP Client): درخواست‌هایی به سمت سرور می‌فرستد و پاسخ دریافت می‌کند
- رایج‌ترین Client روزمره: **Web Browser** (مانند Chrome یا Firefox)

نویسندگان یک مثال عملی می‌زنند: وقتی آدرس `http://www.oreilly.com/index.html` را در مرورگر تایپ می‌کنید، مرورگر یک **HTTP Request** به سرور `www.oreilly.com` ارسال می‌کند. سرور فایل `/index.html` را پیدا کرده و در قالب یک **HTTP Response** — به همراه نوع فایل، طول آن و سایر اطلاعات — برمی‌گرداند. 

> **نکته معماری:** Client و Server با هم «اجزای پایه‌ای World Wide Web» را تشکیل می‌دهند. این یک معماری **Client-Server** خالص است.

# فصل اول — بخش دوم: Resources، MIME Types و URI/URL

## Web Resource چیست؟

نویسندگان تعریف دقیقی از Resource ارائه می‌دهند: **«هر منبعی که محتوای وب تولید کند.»** 

دو نوع Resource وجود دارد:

- **Static Resource:** فایل‌هایی که روی فایل‌سیستم سرور ذخیره شده‌اند — HTML، JPEG، Word، PDF، MP4 و هر فرمت دیگری. اینها ثابت‌اند؛ همیشه همان محتوا را برمی‌گردانند.
- **Dynamic Resource:** برنامه‌هایی که محتوا را **در لحظه** تولید می‌کنند — بر اساس هویت کاربر، زمان روز، داده‌های درخواست‌شده یا اطلاعات پایگاه داده. 

مثال‌های Dynamic Resource که کتاب ذکر می‌کند: تصویر زنده از یک دوربین، سیستم معامله سهام، جستجوی پایگاه داده ملک، یا فروشگاه آنلاین. 

> **نتیجه‌گیری کتاب:** Resource یعنی هر چیزی که منبع محتوا باشد — یک اسپردشیت فروش شرکت، یک Gateway برای جستجوی کتابخانه، یا یک موتور جستجو. 

## MIME Type — برچسب فرمت داده

از آنجا که اینترنت هزاران نوع داده مختلف دارد، HTTP به یک مکانیزم برچسب‌گذاری نیاز داشت. اینجاست که **MIME Type** (Multipurpose Internet Mail Extensions) وارد می‌شود. 

MIME در اصل برای حل مشکل انتقال پیام‌ها بین سیستم‌های ایمیل مختلف طراحی شده بود. HTTP این استاندارد را «قرض گرفت» و برای برچسب‌گذاری محتوای چندرسانه‌ای خود استفاده کرد. 

**ساختار MIME Type:**

```
نوع‌اصلی/نوع‌فرعی
```

| MIME Type | توضیح |
|---|---|
| `text/html` | سند HTML |
| `text/plain` | متن ساده ASCII |
| `image/jpeg` | تصویر JPEG |
| `image/gif` | تصویر GIF |
| `video/quicktime` | فیلم Apple QuickTime |
| `application/vnd.ms-powerpoint` | ارائه PowerPoint |

وقتی Browser یک Object از سرور دریافت می‌کند، به MIME Type نگاه می‌کند تا بفهمد چطور آن را نمایش دهد. اکثر مرورگرها صدها نوع محتوا را می‌شناسند: نمایش تصویر، پارس HTML، پخش صدا از بلندگو، یا راه‌اندازی Plugin خارجی. 

## URI — آدرس پستی اینترنت

هر Resource روی سرور یک **نام منحصربه‌فرد** دارد که به آن **URI (Uniform Resource Identifier)** می‌گویند. نویسندگان URI را به «آدرس پستی اینترنت» تشبیه می‌کنند — به‌طور یکتا یک Resource را در سراسر جهان شناسایی و مکان‌یابی می‌کند. 

URI دو «طعم» دارد:

### URL — مکان دقیق Resource

**URL (Uniform Resource Locator)** رایج‌ترین شکل URI است. URL مکان دقیق یک Resource را روی یک سرور خاص توصیف می‌کند. 

**ساختار استاندارد URL از سه بخش تشکیل شده:** 

```
http://www.joes-hardware.com/specials/saw-blade.gif
  │              │                      │
 Scheme      Server Address         Resource Path
(Protocol)  (آدرس اینترنتی سرور)   (مسیر Resource)
```

- **Scheme:** پروتکل دسترسی (معمولاً `http://`)
- **Server Address:** آدرس اینترنتی سرور (مثل `www.joes-hardware.com`)
- **Resource Path:** مسیر Resource روی سرور (مثل `/specials/saw-blade.gif`)

چند مثال از کتاب: 

| URL | توضیح |
|---|---|
| `http://www.oreilly.com/index.html` | صفحه اصلی O'Reilly |
| `http://www.yahoo.com/images/logo.gif` | لوگوی Yahoo |
| `ftp://joe:tools4u@ftp.joes-hardware.com/locking-pliers.gif` | فایل تصویر با FTP احراز هویت‌شده |

### URN — هویت مستقل از مکان

**URN (Uniform Resource Name)** یک نام یکتا برای یک محتوای خاص است — **مستقل از اینکه آن Resource الان کجا قرار دارد.** 

مثال از کتاب:
```
urn:ietf:rfc:2141
```

این URN سند استاندارد اینترنتی «RFC 2141» را نام‌گذاری می‌کند، حتی اگر آن سند به مکان دیگری منتقل شود یا در چندین جا کپی باشد. 

> **وضعیت فعلی URN:** هنوز در مرحله آزمایشی است و به‌طور گسترده پذیرفته نشده. دلیل اصلی: نیاز به یک زیرساخت پشتیبان برای Resolve کردن مکان Resource دارد که هنوز ایجاد نشده.  بنابراین در عمل و در بقیه کتاب، URI و URL به‌جای هم استفاده می‌شوند. 

## یک نکته معماری مهم

رابطه این مفاهیم را ذهن نگه دارید:

```
URI (مفهوم کلی شناسه)
 ├── URL (شناسه بر اساس مکان) ← مورد استفاده فعلی
 └── URN (شناسه بر اساس نام)  ← هنوز آزمایشی
```

این تمایز بعداً در فصل‌های بعدی کتاب اهمیت پیدا می‌کند، به‌ویژه وقتی درباره Caching و Resource Identity صحبت می‌شود.

# فصل اول — بخش سوم: Transactions، Messages و Connections

## HTTP Transaction چیست؟

یک **HTTP Transaction** از دو بخش تشکیل می‌شود: 

1. **Request Command:** از Client به سمت Server ارسال می‌شود
2. **Response Result:** از Server به سمت Client برمی‌گردد

این تبادل از طریق بلوک‌های فرمت‌شده داده به نام **HTTP Messages** انجام می‌شود. 

## HTTP Methods — دستوراتی که به سرور می‌گویند چه کند

هر HTTP Request Message یک **Method** دارد که به سرور می‌گوید چه عملیاتی انجام دهد. پنج Method رایج که کتاب معرفی می‌کند: 

| HTTP Method | توضیح |
|---|---|
| `GET` | یک Resource نام‌دار را از سرور به Client ارسال کن |
| `PUT` | داده‌های Client را در یک Resource نام‌دار روی سرور ذخیره کن |
| `DELETE` | یک Resource نام‌دار را از سرور حذف کن |
| `POST` | داده‌های Client را به یک Gateway Application سرور بفرست |
| `HEAD` | فقط HTTP Header های Response را برای Resource نام‌دار بفرست |

## Status Codes — نتیجه Transaction را گزارش می‌دهند

هر HTTP Response Message یک **Status Code** سه‌رقمی دارد که به Client می‌گوید درخواست موفق بوده یا نیاز به اقدام دیگری است. 

| Status Code | توضیح |
|---|---|
| `200 OK` | سند با موفقیت برگردانده شد |
| `302 Redirect` | به جای دیگری برو تا Resource را بگیری |
| `404 Not Found` | این Resource پیدا نشد |

یک نکته مهم که کتاب تأکید می‌کند: HTTP علاوه بر Status Code عددی، یک **Reason Phrase** متنی هم می‌فرستد — صرفاً برای توصیف. اما تمام پردازش‌ها فقط بر اساس عدد انجام می‌شوند.  به همین دلیل این چهار Response از نظر HTTP کاملاً یکسان هستند: 

```
200 OK
200 Document attached
200 Success
200 All's cool, dude
```

## Web Pages — مجموعه‌ای از Resources، نه یک Resource

یک نکته معماری مهم که بسیاری از توسعه‌دهندگان نادیده می‌گیرند: یک اپلیکیشن اغلب **چندین HTTP Transaction** برای انجام یک کار صادر می‌کند. 

وقتی مرورگر یک صفحه وب گرافیکی را بارگذاری می‌کند:
1. ابتدا یک Transaction برای دریافت **HTML Skeleton** (ساختار صفحه) انجام می‌دهد
2. سپس برای هر تصویر، هر Graphics Pane و هر Java Applet، یک Transaction **جداگانه** صادر می‌کند
3. این Resource های جاسازی‌شده حتی ممکن است روی **سرورهای مختلف** باشند 

> **نتیجه:** یک «صفحه وب» در واقع یک مجموعه از Resources است، نه یک Resource واحد. 

## ساختار HTTP Messages

HTTP Messages رشته‌های ساده و **Line-Oriented** از کاراکترها هستند. چون متن ساده (Plain Text) هستند نه باینری، خواندن و نوشتن آنها برای انسان آسان است. 

دو نوع Message وجود دارد:
- **Request Message:** از Web Client به Web Server
- **Response Message:** از Server به Client

هر Message از **سه بخش** تشکیل می‌شود: 

```
GET /tools.html HTTP/1.0                    ← Start Line
User-agent: Mozilla/4.75                    ← Header Fields
Host: www.joes-hardware.com                 ← Header Fields
Accept: text/html, image/gif                ← Header Fields
                                            ← Blank Line (پایان Headers)
[بدنه اختیاری]                              ← Body
```

### Start Line
اولین خط Message است. در Request می‌گوید «چه کاری انجام بده»؛ در Response می‌گوید «چه اتفاقی افتاد». 

### Header Fields
صفر یا بیشتر Header Field بعد از Start Line می‌آیند. هر Header از یک نام و یک مقدار تشکیل شده که با دو‌نقطه (`:`) از هم جدا می‌شوند. Headers با یک **Blank Line** تمام می‌شوند. 

### Body
بعد از Blank Line، یک Body اختیاری قرار دارد که می‌تواند هر نوع داده‌ای باشد — برخلاف Start Line و Headers که متنی و ساختاریافته‌اند، Body می‌تواند **داده باینری دلخواه** باشد (تصویر، ویدیو، فایل صوتی، نرم‌افزار). 

## Connections — چطور Messages جابه‌جا می‌شوند؟

### لایه‌بندی شبکه

HTTP یک **Application Layer Protocol** است و جزئیات شبکه را به **TCP/IP** واگذار می‌کند. 

```
HTTP        ← Application Layer
TCP         ← Transport Layer
IP          ← Network Layer
Link Interface ← Data Link Layer
Physical Hardware ← Physical Layer
```

TCP سه تضمین اساسی ارائه می‌دهد: 
- **انتقال بدون خطا** (Error-free data transportation)
- **تحویل مرتب** — داده دقیقاً به همان ترتیبی که فرستاده شده می‌رسد
- **جریان داده بدون قطعه‌بندی** — می‌تواند در هر اندازه‌ای داده ارسال کند

### IP Addresses و Port Numbers

قبل از اینکه HTTP Client بتواند پیامی بفرستد، باید یک **TCP/IP Connection** برقرار کند. برای این کار به دو چیز نیاز دارد: 

- **IP Address:** آدرس سرور (مثل `207.200.83.29`)
- **Port Number:** شماره پورت نرم‌افزار روی سرور

از کجا IP Address و Port Number را پیدا می‌کند؟ از **URL**. 

```
http://207.200.83.29:80/index.html     ← IP Address صریح + Port صریح
http://www.netscape.com:80/index.html  ← Hostname + Port صریح
http://www.netscape.com/index.html     ← Hostname + Port پیش‌فرض (80)
```

وقتی Port در URL وجود نداشته باشد، مقدار پیش‌فرض **80** در نظر گرفته می‌شود. Hostname از طریق **DNS (Domain Name Service)** به IP Address تبدیل می‌شود. 

### مراحل کامل یک HTTP Connection

مراحلی که مرورگر برای نمایش یک صفحه طی می‌کند: 

1. **استخراج Hostname** از URL
2. **تبدیل Hostname به IP Address** از طریق DNS
3. **استخراج Port Number** از URL (یا استفاده از پیش‌فرض ۸۰)
4. **برقراری TCP Connection** به سرور روی آن پورت
5. **ارسال HTTP Request Message** به سرور
6. **خواندن HTTP Response Message** از سرور
7. **بستن Connection**

# فصل اول — بخش چهارم: Protocol Versions و Architectural Components

## نسخه‌های HTTP

کتاب چهار نسخه اصلی HTTP را معرفی می‌کند که هر کدام مشکلات نسخه قبلی را حل کردند: 

### HTTP/0.9
اولین نسخه HTTP، مربوط به سال ۱۹۹۱، بسیار ساده و ابتدایی بود. فقط از Method `GET` پشتیبانی می‌کرد و هیچ MIME Type، HTTP Header یا Version Number ای نداشت. هدف اولیه‌اش فقط انتقال داده HTML بود. 

### HTTP/1.0
اولین نسخه‌ای که به‌طور گسترده استفاده شد. در این نسخه Version Number، HTTP Headers، Method های اضافه و پشتیبانی از MIME Type اضافه شدند. این باعث شد HTTP بتواند انواع داده‌های مختلف را منتقل کند. 

> **نکته:** HTTP/1.0 یک استاندارد رسمی نبود — بیشتر مستندسازی رفتارهای رایجی بود که در عمل استفاده می‌شدند.

### HTTP/1.1
این نسخه اولین استاندارد رسمی HTTP است. روی **بهینه‌سازی Performance**، رفع ابهامات معماری و پشتیبانی از ویژگی‌های پیشرفته‌تر تمرکز کرد. مهم‌ترین چیزی که اضافه کرد **Persistent Connections** بود — Connection ها دیگر بعد از هر Request بسته نمی‌شدند. 

> این نسخه همان چیزی است که بقیه کتاب عمدتاً روی آن تمرکز دارد.

### HTTP-NG (HTTP/2)
نسخه‌ای که در زمان نوشتن کتاب هنوز در مرحله طراحی بود. هدفش بهبود چشمگیر Performance از طریق **Modular Architecture** و **Distributed Object** بود. 

## Architectural Components of the Web

تا اینجا فقط درباره Client و Server صحبت کردیم. اما کتاب تأکید می‌کند که وب از اجزای معماری بیشتری تشکیل شده است. 

### Proxies
یک **Proxy** بین Client و Server قرار می‌گیرد. درخواست‌های HTTP از Client را دریافت کرده و آنها را به سرور می‌فرستد. 

Proxy ها اغلب برای **امنیت** استفاده می‌شوند — می‌توانند همه Traffic را فیلتر کنند. مثلاً در یک محیط کارپوراتیو، Proxy می‌تواند محتوای خطرناک را شناسایی و مسدود کند قبل از اینکه به Client برسد. 

### Caches
یک **Web Cache** یا **Caching Proxy** نوع خاصی از Proxy است. نسخه‌های Local از اسناد پرتقاضا را نگه می‌دارد تا در دفعه بعدی نیازی به دریافت از سرور اصلی نباشد. 

مزیت اصلی: کاهش چشمگیر **Traffic** و **Latency**. به‌جای اینکه هر Client همان محتوا را از سرور دور دریافت کند، Cache نزدیک‌تر آن را تحویل می‌دهد. 

### Gateways
**Gateway** ها سرورهای خاصی هستند که به‌عنوان واسط برای سرورهای دیگر عمل می‌کنند. اغلب برای تبدیل HTTP Traffic به پروتکل دیگری استفاده می‌شوند. 

یک مثال رایج: **HTTP/FTP Gateway** — یک Gateway که درخواست‌های HTTP را دریافت کرده و فایل‌ها را از طریق FTP Fetch می‌کند. از دیدگاه Client، این Gateway فقط یک Web Server معمولی به نظر می‌رسد. 

### Tunnels
**Tunnel** ها اتصالات کوری هستند که داده خام را بین دو Connection انتقال می‌دهند بدون اینکه به محتوا نگاه کنند. 

رایج‌ترین کاربرد: انتقال **SSL Traffic** رمزنگاری‌شده از طریق یک HTTP Connection — به این ترتیب SSL Traffic می‌تواند از میان یک HTTP Firewall عبور کند، حتی اگر Firewall محتوا را نفهمد. 

### Agents
**Agent** ها Client هایی هستند که به‌صورت خودکار و بدون نظارت انسانی درخواست‌های HTTP صادر می‌کنند. 

انواع Agent ها:
- **Web Browser** (آشناترین نوع)
- **Web Spider / Web Crawler:** وب را برای ایجاد ایندکس موتورهای جستجو می‌گردند
- **Web Robot:** وظایف خودکار مختلف انجام می‌دهند

## جمع‌بندی معماری کلی وب

همه این اجزا در کنار هم یک تصویر کامل می‌سازند: 

```
[Agent/Browser]
      │
      │ HTTP Request
      ▼
   [Proxy] ──── [Cache]
      │
      │ HTTP Request (فیلترشده)
      ▼
  [Gateway]
      │
      │ پروتکل دیگر (FTP، SSL، ...)
      ▼
  [Origin Server]
```

نویسندگان این فصل را با یک جمله قدرتمند تمام می‌کنند: **«این پایان شروع است.»**  یعنی همه چیزی که در این فصل گفته شد فقط یک نمای کلی بود — هر کدام از این مفاهیم در فصل‌های بعدی با عمق کامل بررسی می‌شوند.
