## توضیحات
جامع‌ترین مرجع موجود درباره پروتکل HTTP که در سال ۲۰۰۲ توسط O'Reilly منتشر شد و نوشته دیوید گورلی و برایان توتی به همراه سه نویسنده دیگر است. کتاب در ۲۱ فصل و ۶۵۶ صفحه، نه‌تنها HTTP را توضیح می‌دهد بلکه تمام فناوری‌های هسته‌ای وابسته به آن — از مدیریت اتصال TCP، پراکسی و Cache، احراز هویت، HTTPS تا Redirect و Load Balancing — را با دیاگرام‌های دقیق پوشش می‌دهد. این کتاب بیشتر یک «چرا و چطور» است تا یک مرجع header-by-header.

## نظر

- `به دیگران توصیه می‌کنم` : بله

- `نکات مثبت` : کامل‌ترین توضیح HTTP/1.1 که در قالب یک کتاب یافت می‌شود؛ دیاگرام‌های فراوان که مفاهیم انتزاعی پروتکل را ملموس می‌کنند؛ پوشش عمیق موضوعاتی که اکثر کتاب‌های دیگر از آن‌ها عبور می‌کنند مثل Web Robots، Content Negotiation و Internationalization
- `نکات منفی` : HTTP/2 و HTTP/3 را پوشش نمی‌دهد چون قبل از آن‌ها نوشته شده؛ نمونه کدها به Perl و C نوشته شده‌اند که امروز بی‌ربط به نظر می‌رسند؛ برای کسی که فقط دنبال کار با API های مدرن است، عمق کتاب بیش از نیاز است

# فصل اول — بخش اول: HTTP چیست و چرا اهمیت دارد؟

## معرفی و جایگاه HTTP

نویسندگان کتاب با یک ادعای ساده اما قوی شروع می‌کنند: «HTTP زبان مشترک اینترنت مدرن است.»  این جمله را جدی بگیرید. وقتی می‌گویند «زبان مشترک»، منظورشان این است که تمام Web Browsers، Web Servers و Web Applications — بدون استثنا — برای تبادل اطلاعات روی HTTP تکیه می‌کنند. 

نکته‌ای که اینجا مهم است و بسیاری از توسعه‌دهندگان نادیده می‌گیرند: HTTP یک **Application-layer Protocol** است. این یعنی HTTP اصلاً نگران جزئیات شبکه نیست — آن جزئیات را به TCP/IP واگذار کرده.  این تقسیم مسئولیت (Separation of Concerns) در طراحی HTTP، یکی از تصمیمات معماری بنیادی این پروتکل است. 

## HTTP به‌عنوان یک Multimedia Courier

نویسندگان HTTP را «پیک چندرسانه‌ای اینترنت» می‌نامند.  چرا؟ چون HTTP روزانه میلیاردها فایل از انواع مختلف — تصویر JPEG، صفحه HTML، فایل متنی، فیلم MPEG، فایل صوتی WAV و Java Applet — را با سرعت، راحتی و قابلیت اطمینان جابه‌جا می‌کند. 

یک ویژگی حیاتی که کتاب روی آن تأکید می‌کند: HTTP از **Reliable Data-Transmission Protocols** استفاده می‌کند. این تضمین می‌دهد که داده شما در مسیر انتقال خراب یا تغییر نمی‌کند.  برای شما به‌عنوان یک توسعه‌دهنده .NET Core، این یعنی لایه HTTP زیرساخت انتقال را تضمین می‌کند و شما می‌توانید تمام انرژی خود را روی منطق اپلیکیشن متمرکز کنید. 

## Web Clients و Web Servers

کتاب ساختار اصلی را این‌گونه تعریف می‌کند: 

- **Web Server** (HTTP Server): داده‌های اینترنت را نگه می‌دارد و در پاسخ به درخواست‌ها ارائه می‌دهد
- **Web Client** (HTTP Client): درخواست‌هایی به سمت سرور می‌فرستد و پاسخ دریافت می‌کند
- رایج‌ترین Client روزمره: **Web Browser** (مانند Chrome یا Firefox)

نویسندگان یک مثال عملی می‌زنند: وقتی آدرس `http://www.oreilly.com/index.html` را در مرورگر تایپ می‌کنید، مرورگر یک **HTTP Request** به سرور `www.oreilly.com` ارسال می‌کند. سرور فایل `/index.html` را پیدا کرده و در قالب یک **HTTP Response** — به همراه نوع فایل، طول آن و سایر اطلاعات — برمی‌گرداند. 

> **نکته معماری:** Client و Server با هم «اجزای پایه‌ای World Wide Web» را تشکیل می‌دهند. این یک معماری **Client-Server** خالص است.

# فصل اول — بخش دوم: Resources، MIME Types و URI/URL

## Web Resource چیست؟

نویسندگان تعریف دقیقی از Resource ارائه می‌دهند: **«هر منبعی که محتوای وب تولید کند.»** 

دو نوع Resource وجود دارد:

- **Static Resource:** فایل‌هایی که روی فایل‌سیستم سرور ذخیره شده‌اند — HTML، JPEG، Word، PDF، MP4 و هر فرمت دیگری. اینها ثابت‌اند؛ همیشه همان محتوا را برمی‌گردانند.
- **Dynamic Resource:** برنامه‌هایی که محتوا را **در لحظه** تولید می‌کنند — بر اساس هویت کاربر، زمان روز، داده‌های درخواست‌شده یا اطلاعات پایگاه داده. 

مثال‌های Dynamic Resource که کتاب ذکر می‌کند: تصویر زنده از یک دوربین، سیستم معامله سهام، جستجوی پایگاه داده ملک، یا فروشگاه آنلاین. 

> **نتیجه‌گیری کتاب:** Resource یعنی هر چیزی که منبع محتوا باشد — یک اسپردشیت فروش شرکت، یک Gateway برای جستجوی کتابخانه، یا یک موتور جستجو. 

## MIME Type — برچسب فرمت داده

از آنجا که اینترنت هزاران نوع داده مختلف دارد، HTTP به یک مکانیزم برچسب‌گذاری نیاز داشت. اینجاست که **MIME Type** (Multipurpose Internet Mail Extensions) وارد می‌شود. 

MIME در اصل برای حل مشکل انتقال پیام‌ها بین سیستم‌های ایمیل مختلف طراحی شده بود. HTTP این استاندارد را «قرض گرفت» و برای برچسب‌گذاری محتوای چندرسانه‌ای خود استفاده کرد. 

**ساختار MIME Type:**

```
نوع‌اصلی/نوع‌فرعی
```

| MIME Type | توضیح |
|---|---|
| `text/html` | سند HTML |
| `text/plain` | متن ساده ASCII |
| `image/jpeg` | تصویر JPEG |
| `image/gif` | تصویر GIF |
| `video/quicktime` | فیلم Apple QuickTime |
| `application/vnd.ms-powerpoint` | ارائه PowerPoint |

وقتی Browser یک Object از سرور دریافت می‌کند، به MIME Type نگاه می‌کند تا بفهمد چطور آن را نمایش دهد. اکثر مرورگرها صدها نوع محتوا را می‌شناسند: نمایش تصویر، پارس HTML، پخش صدا از بلندگو، یا راه‌اندازی Plugin خارجی. 

## URI — آدرس پستی اینترنت

هر Resource روی سرور یک **نام منحصربه‌فرد** دارد که به آن **URI (Uniform Resource Identifier)** می‌گویند. نویسندگان URI را به «آدرس پستی اینترنت» تشبیه می‌کنند — به‌طور یکتا یک Resource را در سراسر جهان شناسایی و مکان‌یابی می‌کند. 

URI دو «طعم» دارد:

### URL — مکان دقیق Resource

**URL (Uniform Resource Locator)** رایج‌ترین شکل URI است. URL مکان دقیق یک Resource را روی یک سرور خاص توصیف می‌کند. 

**ساختار استاندارد URL از سه بخش تشکیل شده:** 

```
http://www.joes-hardware.com/specials/saw-blade.gif
  │              │                      │
 Scheme      Server Address         Resource Path
(Protocol)  (آدرس اینترنتی سرور)   (مسیر Resource)
```

- **Scheme:** پروتکل دسترسی (معمولاً `http://`)
- **Server Address:** آدرس اینترنتی سرور (مثل `www.joes-hardware.com`)
- **Resource Path:** مسیر Resource روی سرور (مثل `/specials/saw-blade.gif`)

چند مثال از کتاب: 

| URL | توضیح |
|---|---|
| `http://www.oreilly.com/index.html` | صفحه اصلی O'Reilly |
| `http://www.yahoo.com/images/logo.gif` | لوگوی Yahoo |
| `ftp://joe:tools4u@ftp.joes-hardware.com/locking-pliers.gif` | فایل تصویر با FTP احراز هویت‌شده |

### URN — هویت مستقل از مکان

**URN (Uniform Resource Name)** یک نام یکتا برای یک محتوای خاص است — **مستقل از اینکه آن Resource الان کجا قرار دارد.** 

مثال از کتاب:
```
urn:ietf:rfc:2141
```

این URN سند استاندارد اینترنتی «RFC 2141» را نام‌گذاری می‌کند، حتی اگر آن سند به مکان دیگری منتقل شود یا در چندین جا کپی باشد. 

> **وضعیت فعلی URN:** هنوز در مرحله آزمایشی است و به‌طور گسترده پذیرفته نشده. دلیل اصلی: نیاز به یک زیرساخت پشتیبان برای Resolve کردن مکان Resource دارد که هنوز ایجاد نشده.  بنابراین در عمل و در بقیه کتاب، URI و URL به‌جای هم استفاده می‌شوند. 

## یک نکته معماری مهم

رابطه این مفاهیم را ذهن نگه دارید:

```
URI (مفهوم کلی شناسه)
 ├── URL (شناسه بر اساس مکان) ← مورد استفاده فعلی
 └── URN (شناسه بر اساس نام)  ← هنوز آزمایشی
```

این تمایز بعداً در فصل‌های بعدی کتاب اهمیت پیدا می‌کند، به‌ویژه وقتی درباره Caching و Resource Identity صحبت می‌شود.

# فصل اول — بخش سوم: Transactions، Messages و Connections

## HTTP Transaction چیست؟

یک **HTTP Transaction** از دو بخش تشکیل می‌شود: 

1. **Request Command:** از Client به سمت Server ارسال می‌شود
2. **Response Result:** از Server به سمت Client برمی‌گردد

این تبادل از طریق بلوک‌های فرمت‌شده داده به نام **HTTP Messages** انجام می‌شود. 

## HTTP Methods — دستوراتی که به سرور می‌گویند چه کند

هر HTTP Request Message یک **Method** دارد که به سرور می‌گوید چه عملیاتی انجام دهد. پنج Method رایج که کتاب معرفی می‌کند: 

| HTTP Method | توضیح |
|---|---|
| `GET` | یک Resource نام‌دار را از سرور به Client ارسال کن |
| `PUT` | داده‌های Client را در یک Resource نام‌دار روی سرور ذخیره کن |
| `DELETE` | یک Resource نام‌دار را از سرور حذف کن |
| `POST` | داده‌های Client را به یک Gateway Application سرور بفرست |
| `HEAD` | فقط HTTP Header های Response را برای Resource نام‌دار بفرست |

## Status Codes — نتیجه Transaction را گزارش می‌دهند

هر HTTP Response Message یک **Status Code** سه‌رقمی دارد که به Client می‌گوید درخواست موفق بوده یا نیاز به اقدام دیگری است. 

| Status Code | توضیح |
|---|---|
| `200 OK` | سند با موفقیت برگردانده شد |
| `302 Redirect` | به جای دیگری برو تا Resource را بگیری |
| `404 Not Found` | این Resource پیدا نشد |

یک نکته مهم که کتاب تأکید می‌کند: HTTP علاوه بر Status Code عددی، یک **Reason Phrase** متنی هم می‌فرستد — صرفاً برای توصیف. اما تمام پردازش‌ها فقط بر اساس عدد انجام می‌شوند.  به همین دلیل این چهار Response از نظر HTTP کاملاً یکسان هستند: 

```
200 OK
200 Document attached
200 Success
200 All's cool, dude
```

## Web Pages — مجموعه‌ای از Resources، نه یک Resource

یک نکته معماری مهم که بسیاری از توسعه‌دهندگان نادیده می‌گیرند: یک اپلیکیشن اغلب **چندین HTTP Transaction** برای انجام یک کار صادر می‌کند. 

وقتی مرورگر یک صفحه وب گرافیکی را بارگذاری می‌کند:
1. ابتدا یک Transaction برای دریافت **HTML Skeleton** (ساختار صفحه) انجام می‌دهد
2. سپس برای هر تصویر، هر Graphics Pane و هر Java Applet، یک Transaction **جداگانه** صادر می‌کند
3. این Resource های جاسازی‌شده حتی ممکن است روی **سرورهای مختلف** باشند 

> **نتیجه:** یک «صفحه وب» در واقع یک مجموعه از Resources است، نه یک Resource واحد. 

## ساختار HTTP Messages

HTTP Messages رشته‌های ساده و **Line-Oriented** از کاراکترها هستند. چون متن ساده (Plain Text) هستند نه باینری، خواندن و نوشتن آنها برای انسان آسان است. 

دو نوع Message وجود دارد:
- **Request Message:** از Web Client به Web Server
- **Response Message:** از Server به Client

هر Message از **سه بخش** تشکیل می‌شود: 

```
GET /tools.html HTTP/1.0                    ← Start Line
User-agent: Mozilla/4.75                    ← Header Fields
Host: www.joes-hardware.com                 ← Header Fields
Accept: text/html, image/gif                ← Header Fields
                                            ← Blank Line (پایان Headers)
[بدنه اختیاری]                              ← Body
```

### Start Line
اولین خط Message است. در Request می‌گوید «چه کاری انجام بده»؛ در Response می‌گوید «چه اتفاقی افتاد». 

### Header Fields
صفر یا بیشتر Header Field بعد از Start Line می‌آیند. هر Header از یک نام و یک مقدار تشکیل شده که با دو‌نقطه (`:`) از هم جدا می‌شوند. Headers با یک **Blank Line** تمام می‌شوند. 

### Body
بعد از Blank Line، یک Body اختیاری قرار دارد که می‌تواند هر نوع داده‌ای باشد — برخلاف Start Line و Headers که متنی و ساختاریافته‌اند، Body می‌تواند **داده باینری دلخواه** باشد (تصویر، ویدیو، فایل صوتی، نرم‌افزار). 

## Connections — چطور Messages جابه‌جا می‌شوند؟

### لایه‌بندی شبکه

HTTP یک **Application Layer Protocol** است و جزئیات شبکه را به **TCP/IP** واگذار می‌کند. 

```
HTTP        ← Application Layer
TCP         ← Transport Layer
IP          ← Network Layer
Link Interface ← Data Link Layer
Physical Hardware ← Physical Layer
```

TCP سه تضمین اساسی ارائه می‌دهد: 
- **انتقال بدون خطا** (Error-free data transportation)
- **تحویل مرتب** — داده دقیقاً به همان ترتیبی که فرستاده شده می‌رسد
- **جریان داده بدون قطعه‌بندی** — می‌تواند در هر اندازه‌ای داده ارسال کند

### IP Addresses و Port Numbers

قبل از اینکه HTTP Client بتواند پیامی بفرستد، باید یک **TCP/IP Connection** برقرار کند. برای این کار به دو چیز نیاز دارد: 

- **IP Address:** آدرس سرور (مثل `207.200.83.29`)
- **Port Number:** شماره پورت نرم‌افزار روی سرور

از کجا IP Address و Port Number را پیدا می‌کند؟ از **URL**. 

```
http://207.200.83.29:80/index.html     ← IP Address صریح + Port صریح
http://www.netscape.com:80/index.html  ← Hostname + Port صریح
http://www.netscape.com/index.html     ← Hostname + Port پیش‌فرض (80)
```

وقتی Port در URL وجود نداشته باشد، مقدار پیش‌فرض **80** در نظر گرفته می‌شود. Hostname از طریق **DNS (Domain Name Service)** به IP Address تبدیل می‌شود. 

### مراحل کامل یک HTTP Connection

مراحلی که مرورگر برای نمایش یک صفحه طی می‌کند: 

1. **استخراج Hostname** از URL
2. **تبدیل Hostname به IP Address** از طریق DNS
3. **استخراج Port Number** از URL (یا استفاده از پیش‌فرض ۸۰)
4. **برقراری TCP Connection** به سرور روی آن پورت
5. **ارسال HTTP Request Message** به سرور
6. **خواندن HTTP Response Message** از سرور
7. **بستن Connection**

# فصل اول — بخش چهارم: Protocol Versions و Architectural Components

## نسخه‌های HTTP

کتاب چهار نسخه اصلی HTTP را معرفی می‌کند که هر کدام مشکلات نسخه قبلی را حل کردند: 

### HTTP/0.9
اولین نسخه HTTP، مربوط به سال ۱۹۹۱، بسیار ساده و ابتدایی بود. فقط از Method `GET` پشتیبانی می‌کرد و هیچ MIME Type، HTTP Header یا Version Number ای نداشت. هدف اولیه‌اش فقط انتقال داده HTML بود. 

### HTTP/1.0
اولین نسخه‌ای که به‌طور گسترده استفاده شد. در این نسخه Version Number، HTTP Headers، Method های اضافه و پشتیبانی از MIME Type اضافه شدند. این باعث شد HTTP بتواند انواع داده‌های مختلف را منتقل کند. 

> **نکته:** HTTP/1.0 یک استاندارد رسمی نبود — بیشتر مستندسازی رفتارهای رایجی بود که در عمل استفاده می‌شدند.

### HTTP/1.1
این نسخه اولین استاندارد رسمی HTTP است. روی **بهینه‌سازی Performance**، رفع ابهامات معماری و پشتیبانی از ویژگی‌های پیشرفته‌تر تمرکز کرد. مهم‌ترین چیزی که اضافه کرد **Persistent Connections** بود — Connection ها دیگر بعد از هر Request بسته نمی‌شدند. 

> این نسخه همان چیزی است که بقیه کتاب عمدتاً روی آن تمرکز دارد.

### HTTP-NG (HTTP/2)
نسخه‌ای که در زمان نوشتن کتاب هنوز در مرحله طراحی بود. هدفش بهبود چشمگیر Performance از طریق **Modular Architecture** و **Distributed Object** بود. 

## Architectural Components of the Web

تا اینجا فقط درباره Client و Server صحبت کردیم. اما کتاب تأکید می‌کند که وب از اجزای معماری بیشتری تشکیل شده است. 

### Proxies
یک **Proxy** بین Client و Server قرار می‌گیرد. درخواست‌های HTTP از Client را دریافت کرده و آنها را به سرور می‌فرستد. 

Proxy ها اغلب برای **امنیت** استفاده می‌شوند — می‌توانند همه Traffic را فیلتر کنند. مثلاً در یک محیط کارپوراتیو، Proxy می‌تواند محتوای خطرناک را شناسایی و مسدود کند قبل از اینکه به Client برسد. 

### Caches
یک **Web Cache** یا **Caching Proxy** نوع خاصی از Proxy است. نسخه‌های Local از اسناد پرتقاضا را نگه می‌دارد تا در دفعه بعدی نیازی به دریافت از سرور اصلی نباشد. 

مزیت اصلی: کاهش چشمگیر **Traffic** و **Latency**. به‌جای اینکه هر Client همان محتوا را از سرور دور دریافت کند، Cache نزدیک‌تر آن را تحویل می‌دهد. 

### Gateways
**Gateway** ها سرورهای خاصی هستند که به‌عنوان واسط برای سرورهای دیگر عمل می‌کنند. اغلب برای تبدیل HTTP Traffic به پروتکل دیگری استفاده می‌شوند. 

یک مثال رایج: **HTTP/FTP Gateway** — یک Gateway که درخواست‌های HTTP را دریافت کرده و فایل‌ها را از طریق FTP Fetch می‌کند. از دیدگاه Client، این Gateway فقط یک Web Server معمولی به نظر می‌رسد. 

### Tunnels
**Tunnel** ها اتصالات کوری هستند که داده خام را بین دو Connection انتقال می‌دهند بدون اینکه به محتوا نگاه کنند. 

رایج‌ترین کاربرد: انتقال **SSL Traffic** رمزنگاری‌شده از طریق یک HTTP Connection — به این ترتیب SSL Traffic می‌تواند از میان یک HTTP Firewall عبور کند، حتی اگر Firewall محتوا را نفهمد. 

### Agents
**Agent** ها Client هایی هستند که به‌صورت خودکار و بدون نظارت انسانی درخواست‌های HTTP صادر می‌کنند. 

انواع Agent ها:
- **Web Browser** (آشناترین نوع)
- **Web Spider / Web Crawler:** وب را برای ایجاد ایندکس موتورهای جستجو می‌گردند
- **Web Robot:** وظایف خودکار مختلف انجام می‌دهند

## جمع‌بندی معماری کلی وب

همه این اجزا در کنار هم یک تصویر کامل می‌سازند: 

```
[Agent/Browser]
      │
      │ HTTP Request
      ▼
   [Proxy] ──── [Cache]
      │
      │ HTTP Request (فیلترشده)
      ▼
  [Gateway]
      │
      │ پروتکل دیگر (FTP، SSL، ...)
      ▼
  [Origin Server]
```

نویسندگان این فصل را با یک جمله قدرتمند تمام می‌کنند: **«این پایان شروع است.»**  یعنی همه چیزی که در این فصل گفته شد فقط یک نمای کلی بود — هر کدام از این مفاهیم در فصل‌های بعدی با عمق کامل بررسی می‌شوند.

# فصل دوم — بخش اول: URLs و ناوبری در منابع اینترنت

## چرا URL اهمیت دارد؟

قبل از URL، دسترسی به منابع اینترنت یک کار پیچیده بود. کاربران باید پروتکل‌های مختلف را می‌دانستند، آدرس‌های متفاوت را حفظ می‌کردند و برای هر سرویس رویکرد متفاوتی داشتند. 

URL این مشکل را حل کرد: یک **فرمت یکپارچه و استاندارد** که به کاربر و نرم‌افزار می‌گوید یک Resource کجاست و چطور باید به آن دسترسی داشت. نویسندگان URL را «نقطه شروع ناوبری در اینترنت» می‌نامند. 

## ساختار کامل URL Syntax

کتاب ساختار کامل URL را به این شکل تعریف می‌کند: 

```
scheme://user:password@host:port/path;params?query#frag
```

هر بخش را جداگانه بررسی می‌کنیم:

### Scheme — «چطور دسترسی داشته باشیم»
اولین بخش URL است و پروتکل دسترسی را مشخص می‌کند. **Case-Insensitive** است یعنی `HTTP` و `http` یکسان هستند. با `://` از بقیه URL جدا می‌شود. 

مثال: `http`، `ftp`، `mailto`، `rtsp`

### Host و Port — «کجا برویم»
**Host** آدرس سرور را مشخص می‌کند — می‌تواند Hostname باشد (مثل `www.joes-hardware.com`) یا IP Address مستقیم (مثل `207.200.83.29`). 

**Port** شماره پورت TCP است. اگر حذف شود، مقدار پیش‌فرض پروتکل استفاده می‌شود — برای HTTP پیش‌فرض **80** است. 

### User و Password — «احراز هویت»
برخی سرورها قبل از دسترسی به داده نیاز به احراز هویت دارند. اگر Username بدون Password نوشته شود، مرورگر Password را درخواست می‌کند. اگر هیچ‌کدام نباشند، بعضی Client ها Username پیش‌فرض `anonymous` را ارسال می‌کنند. 

```
ftp://joe:tools4u@ftp.joes-hardware.com/locking-pliers.gif
      ───  ──────
      User Password
```

### Path — «چه Resource ای می‌خواهیم»
مسیر محلی Resource روی سرور است. از `/` شروع می‌شود و با `/` بخش‌هایش از هم جدا می‌شوند. 

```
/specials/saw-blade.gif
```

### Parameters — «اطلاعات ورودی اضافه»
بعضی پروتکل‌ها به اطلاعات ورودی بیشتری نیاز دارند. Parameters با `;` از Path جدا می‌شوند و می‌توانند چندتایی باشند. 

```
http://www.joes-hardware.com/hammers;sale=false/index.html;graphics=true
```

در این مثال دو Parameter داریم:
- `sale=false` برای بخش `hammers`
- `graphics=true` برای `index.html`

### Query String — «فیلتر کردن داده»
برای ارسال پارامترهای Query به Gateway Application ها (مثل پایگاه داده یا موتور جستجو) استفاده می‌شود. با `?` شروع می‌شود و جفت‌های `key=value` با `&` از هم جدا می‌شوند. 

```
http://www.joes-hardware.com/inventory-check.cgi?item=12731&color=blue
```

### Fragment — «بخش خاصی از Resource»
برای اشاره به یک بخش یا قسمت خاص از Resource استفاده می‌شود. با `#` شروع می‌شود. 

```
http://www.joes-hardware.com/tools.html#drills
```

> **نکته مهم معماری:** Fragment **هرگز** به سرور ارسال نمی‌شود. Browser آن را برای خودش نگه می‌دارد و بعد از دریافت کل سند، به آن بخش از صفحه Scroll می‌کند. 

## URL Shortcuts — میان‌برهای URL

کتاب دو نوع Shortcut معرفی می‌کند که Browser ها از آنها پشتیبانی می‌کنند:

### Relative URLs
به‌جای نوشتن آدرس کامل، می‌توان آدرس نسبی نوشت. Browser آدرس کامل را با ترکیب **Base URL** (آدرس صفحه فعلی) و **Relative URL** می‌سازد. 

مثال: اگر در صفحه `http://www.joes-hardware.com/tools.html` باشیم:

```html
<a href="./hammers.html">Hammers</a>
```

Browser این را به آدرس کامل تبدیل می‌کند:
```
http://www.joes-hardware.com/hammers.html
```

فرآیند تبدیل Relative URL به Absolute URL را **Resolving** می‌گویند. 

### Expandomatic URLs
Browser ها می‌توانند URL را به‌صورت خودکار کامل کنند — بدون اینکه کاربر آدرس کامل تایپ کند. 

دو نوع Auto-Expansion وجود دارد:
- **Hostname Expansion:** اگر `yahoo` تایپ کنید، Browser اضافه می‌کند `www.` و `.com` تا `www.yahoo.com` بشود
- **History Expansion:** Browser تاریخچه بازدیدها را چک می‌کند و URL های مشابه پیشنهاد می‌دهد

> **هشدار کتاب:** این رفتار Auto-Expansion در Proxy ها می‌تواند مشکل‌ساز باشد، چون Proxy URL اصلی تایپ‌شده را می‌بیند نه URL کامل‌شده را.

# فصل دوم — بخش دوم: کاراکترهای خاص و انواع Scheme ها

## مشکل کاراکترهای خاص در URL

URL باید روی **همه پلتفرم‌ها و پروتکل‌ها** قابل استفاده باشد. این یعنی باید هم **قابل حمل (Portable)** باشد و هم **کامل** — بتواند هر Resource ای را نام‌گذاری کند. 

این دو هدف با هم در تضاد هستند. چرا؟ چون سیستم‌های مختلف کاراکترهای مختلفی را مجاز می‌دانند. برای حل این مشکل، طراحان URL یک مکانیزم **Escape** تعریف کردند.

## URL Character Set — کاراکترهای مجاز

URL از **US-ASCII** به‌عنوان مجموعه کاراکتر پایه استفاده می‌کند. دلیلش سادگی و سازگاری با سیستم‌های قدیمی است. 

کاراکترها در URL به سه دسته تقسیم می‌شوند: 

| دسته | توضیح | مثال |
|---|---|---|
| **Unreserved Characters** | همیشه مجاز، نیازی به Encode ندارند | `A-Z`، `a-z`، `0-9`، `-`، `_`، `.`، `~` |
| **Reserved Characters** | معنای خاص دارند؛ اگر به‌عنوان داده استفاده شوند باید Encode شوند | `/`، `?`، `#`، `:`، `@`، `&`، `=` |
| **Unsafe Characters** | ممکن است در انتقال تغییر کنند یا حذف شوند | فاصله، `<`، `>`، `{`، `}`، `|` |

## Percent Encoding — راه‌حل کاراکترهای غیرمجاز

هر کاراکتری که در URL مجاز نباشد باید با **Percent Encoding** کدگذاری شود. فرمت آن ساده است: 

```
% + دو رقم هگزادسیمال از کد ASCII کاراکتر
```

چند مثال رایج: 

| کاراکتر | Encoded |
|---|---|
| فاصله (Space) | `%20` |
| `%` | `%25` |
| `/` (به‌عنوان داده) | `%2F` |
| `@` (به‌عنوان داده) | `%40` |

مثال عملی:
```
http://www.joes-hardware.com/more%20tools.html
```
این URL در واقع فایلی به نام `more tools.html` (با فاصله) را درخواست می‌کند. 

> **نکته مهم برای توسعه‌دهنده:** در .NET Core متد `Uri.EscapeDataString()` برای Encode کردن و `Uri.UnescapeDataString()` برای Decode کردن استفاده می‌شود. هرگز URL را دستی Encode نکنید — جزئیاتی مثل کاراکترهای Reserved که نباید Encode شوند، به‌راحتی فراموش می‌شوند.

## انواع Scheme ها

کتاب مهم‌ترین URL Scheme های موجود را معرفی می‌کند. هر Scheme قوانین و ساختار خاص خودش را دارد: 

### `http` — پرکاربردترین Scheme

```
http://host:port/path?query#frag
```

Port پیش‌فرض: **80**. اگر Port نباشد، همان ۸۰ فرض می‌شود. 

### `https` — نسخه امن HTTP

```
https://host:port/path?query#frag
```

Port پیش‌فرض: **443**. دقیقاً مثل `http` است اما از **SSL/TLS** برای رمزنگاری Connection استفاده می‌کند. 

### `ftp` — File Transfer Protocol

```
ftp://user:password@host:port/path
```

برای دانلود و آپلود فایل روی FTP Server استفاده می‌شود. Port پیش‌فرض: **21**. 

### `mailto` — ارسال ایمیل

```
mailto:addr-spec
```

مثال:
```
mailto:joe@joes-hardware.com
```

این Scheme کمی با بقیه فرق دارد — به یک Resource روی شبکه اشاره نمی‌کند بلکه یک **آدرس ایمیل** است. 

### `file` — فایل محلی

```
file://host/path
```

برای دسترسی به فایل‌های محلی روی همان دستگاه استفاده می‌شود. اگر `host` حذف شود، همان دستگاه محلی فرض می‌شود: 

```
file:///C:/Documents/presentation.ppt
```

### `rtsp` و `rtspu` — Streaming Media

```
rtsp://host:port/path
rtspu://host:port/path
```

برای **Real Time Streaming Protocol** استفاده می‌شود. `rtsp` از TCP و `rtspu` از UDP استفاده می‌کند. 

### `news` — Usenet Newsgroup

```
news:newsgroup
news:message-id
```

برای دسترسی به Usenet Newsgroup ها. یک ویژگی جالب: `news` URL اطلاعات کافی برای مکان‌یابی Resource ندارد — نمی‌گوید از کدام سرور باید خواند. 

### `telnet` — دسترسی به سرویس‌های تعاملی

```
telnet://user:password@host:port/
```

برای ارتباط با سرویس‌های Interactive روی شبکه استفاده می‌شود. 

## آینده URL — از URL به URN

کتاب در پایان این فصل یک مشکل بنیادی URL را مطرح می‌کند: **URL به مکان وابسته است.** 

اگر یک Resource جابه‌جا شود، URL قدیمی دیگر کار نمی‌کند و همه Linkهایی که به آن اشاره می‌کردند «شکسته» می‌شوند. این مشکل معروف به **Link Rot** است.

راه‌حل پیشنهادی: **URN (Uniform Resource Name)** — یک نام ثابت برای Resource که مستقل از مکان آن است. با URN می‌توان Resource را به هر جایی منتقل کرد بدون اینکه Link های موجود بشکنند. 

اما چرا هنوز URN جایگزین URL نشده؟ کتاب دلیل را صریح بیان می‌کند: برای کار کردن URN به یک **Resolution Infrastructure** نیاز است — یک سیستم که بتواند نام URN را به مکان فعلی Resource تبدیل کند. ساخت و پیاده‌سازی چنین زیرساختی در مقیاس اینترنت بسیار پیچیده است.

# فصل سوم — بخش اول: ساختار و جریان HTTP Messages

## Messages چگونه جریان دارند؟

نویسندگان فصل سوم را با یک استعاره شروع می‌کنند: HTTP Messages مثل **آب در لوله** جریان دارند. اصطلاح **Inbound** و **Outbound** برای توصیف جهت جریان Message استفاده می‌شود. 

- **Inbound:** Message به سمت Origin Server حرکت می‌کند
- **Outbound:** Message از Origin Server به سمت User Agent (Browser) برمی‌گردد

یک قانون مهم: **همه Messages — چه Request چه Response — Downstream جریان دارند.** یعنی همه Message ها از فرستنده به سمت گیرنده حرکت می‌کنند، صرف‌نظر از اینکه Request باشند یا Response. 

## دو نوع Message

HTTP دقیقاً دو نوع Message دارد و نه چیز دیگری: 

- **Request Message:** از Client به Server — درخواست انجام یک عملیات
- **Response Message:** از Server به Client — نتیجه آن عملیات

## ساختار کامل یک Message

هر HTTP Message از سه بخش اصلی تشکیل شده: 

```
<start-line>          ← یک خط — توصیف Message
<headers>             ← صفر یا بیشتر — متادیتا
<blank line>          ← یک خط خالی اجباری (CRLF)
<body>                ← اختیاری — داده واقعی
```

### ساختار Request Message:
```
GET /specials/saw-blade.gif HTTP/1.0    ← Request Line
Host: www.joes-hardware.com            ← Header
Accept: image/*                        ← Header
                                       ← Blank Line
                                       ← (Body ندارد)
```

### ساختار Response Message:
```
HTTP/1.0 200 OK                        ← Response Line (Status Line)
Content-type: image/gif                ← Header
Content-length: 8572                   ← Header
                                       ← Blank Line
[binary content of image]              ← Body
```

## Start Line — اولین خط هر Message

### Request Line
در Request Message، Start Line را **Request Line** می‌نامند. از سه بخش تشکیل شده که با **فاصله** از هم جدا می‌شوند: 

```
Method   Request-URL   HTTP-Version
  │           │              │
 GET    /tools.html      HTTP/1.0
```

- **Method:** عملیاتی که سرور باید انجام دهد
- **Request-URL:** آدرس کامل یا مسیر محلی Resource
- **HTTP-Version:** نسخه پروتکل که فرستنده از آن استفاده می‌کند

### Status Line (Response Line)
در Response Message، Start Line را **Status Line** می‌نامند: 

```
HTTP-Version   Status-Code   Reason-Phrase
     │              │               │
 HTTP/1.0          200             OK
```

- **HTTP-Version:** نسخه‌ای که سرور پشتیبانی می‌کند
- **Status-Code:** عدد سه‌رقمی نتیجه عملیات
- **Reason-Phrase:** توضیح متنی Status Code — فقط برای انسان، نه برای ماشین

## Headers — متادیتای Message

Header ها اطلاعات اضافی درباره Request یا Response فراهم می‌کنند. هر Header یک **Name-Value Pair** است که با `:` از هم جدا می‌شوند: 

```
Content-type: text/html
Content-length: 403
```

**قوانین مهم Header ها:**

- هر Header باید در **یک خط** باشد
- اگر یک Header خط بعدی را نیاز داشت، می‌توان با **فاصله یا Tab** در خط بعدی ادامه داد (به این **Folding** می‌گویند)
- Headers با یک **Blank Line** تمام می‌شوند
- **Header Name ها Case-Insensitive هستند** — `Content-Type` و `content-type` یکسان‌اند 

### دسته‌بندی Header ها

کتاب Header ها را به پنج دسته تقسیم می‌کند: 

| دسته | توضیح | مثال |
|---|---|---|
| **General Headers** | هم در Request هم در Response استفاده می‌شوند | `Date`، `Connection` |
| **Request Headers** | فقط در Request Message | `Host`، `Accept`، `User-Agent` |
| **Response Headers** | فقط در Response Message | `Server`، `WWW-Authenticate` |
| **Entity Headers** | اطلاعات درباره Body | `Content-Type`، `Content-Length` |
| **Extension Headers** | Header های سفارشی که در استاندارد نیستند | هر `X-Custom-Header` |

## Body — داده واقعی

Body بخش اختیاری Message است که داده واقعی را حمل می‌کند. 

- **Request Body:** داده‌ای که Client به سرور می‌فرستد — مثل داده فرم در `POST`
- **Response Body:** محتوایی که سرور به Client برمی‌گرداند — HTML، تصویر، JSON و...

یک تفاوت اساسی با Start Line و Headers: Body می‌تواند **داده باینری دلخواه** باشد. تصویر، ویدیو، فایل صوتی، نرم‌افزار — هیچ محدودیتی در نوع داده وجود ندارد. 

> **نکته Performance:** اگر Body داده باینری دارد، **هرگز** نباید آن را به‌عنوان متن parse کنید. در .NET Core برای خواندن Body به‌صورت Stream از `Request.Body` استفاده کنید تا از کپی غیرضروری داده در حافظه جلوگیری شود.

# فصل سوم — بخش اول: ساختار HTTP Messages

## جریان Messages

کتاب با یک تعریف دقیق شروع می‌کند: HTTP Message ها **بلوک‌های پایه‌ای داده** هستند که بین HTTP Application ها ردوبدل می‌شوند. 

پیام‌ها در یک جهت مشخص **جاری** می‌شوند — نویسندگان عمداً از کلمه «flow» استفاده می‌کنند:

- **Inbound:** به سمت Origin Server
- **Outbound:** از Origin Server به سمت Client (بعد از پردازش)
- **Downstream:** صرف‌نظر از Request یا Response، همه پیام‌ها به سمت Downstream جاری می‌شوند

> **نکته معماری:** Sender همیشه نسبت به Receiver در Upstream قرار دارد. این مفهوم بعداً در فصل Proxies و Caching اهمیت پیدا می‌کند. 

## اجزای یک HTTP Message

هر HTTP Message از سه بخش تشکیل می‌شود: 

```
GET /specials/saw-blade.gif HTTP/1.0    ← Start Line
Host: www.joes-hardware.com             ← Headers
Accept: text/*                          ← Headers
                                        ← Blank Line (CRLF)

```

### Start Line
در **Request Message** سه چیز مشخص می‌کند: Method، Request URL و نسخه HTTP. 

```
GET /specials/saw-blade.gif HTTP/1.0
─── ─────────────────────── ────────
Method      Request URL      Version
```

در **Response Message** سه چیز مشخص می‌کند: نسخه HTTP، Status Code و Reason Phrase. 

```
HTTP/1.0 200 OK
──────── ─── ──
Version  Code Reason Phrase
```

### Header Fields
صفر یا بیشتر Header به شکل `Name: Value` بعد از Start Line می‌آیند. Headers با یک **Blank Line** (CRLF خالی) تمام می‌شوند. 

کتاب Headers را به پنج دسته تقسیم می‌کند:

| دسته | توضیح |
|---|---|
| **General Headers** | هم در Request و هم در Response استفاده می‌شوند |
| **Request Headers** | فقط در Request — اطلاعات بیشتر درباره درخواست |
| **Response Headers** | فقط در Response — اطلاعات بیشتر درباره پاسخ |
| **Entity Headers** | اطلاعات درباره Body (طول، نوع محتوا و ...) |
| **Extension Headers** | Header های غیراستاندارد که توسط توسعه‌دهنده تعریف می‌شوند |

یک Header می‌تواند به چند خط تقسیم شود اگر خط بعدی با **فاصله یا Tab** شروع شود: 

```
HTTP/1.0 200 OK
Content-Type: image/gif
Content-Length: 8572
Server: Test Server
    Version 1.0
```

### Body
بعد از Blank Line قرار می‌گیرد. می‌تواند هر نوع داده‌ای باشد — متن، HTML، تصویر، ویدیو، JSON، XML یا هر فرمت دیگری. 

در Request، Body داده‌ای است که به سرور ارسال می‌شود (مثلاً داده‌های یک Form). در Response، Body محتوای درخواست‌شده است.

## HTTP Methods — جزئیات کامل

### GET
رایج‌ترین Method. از سرور می‌خواهد یک Resource را برگرداند. Body ندارد. 

```
GET /seasonal/index-fall.html HTTP/1.1
Host: www.joes-hardware.com
Accept: *
```

پاسخ سرور:
```
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 617

<HTML>
<HEAD><TITLE>Joe's Hardware - Autumn</TITLE></HEAD>
...
```

### HEAD
دقیقاً مثل GET است اما سرور **فقط Headers را برمی‌گرداند، نه Body را.** 

کاربردهای اصلی:
- بررسی وجود یک Resource بدون دانلود محتوا
- بررسی تغییر Resource از آخرین بار (برای Caching)
- بررسی نوع و اندازه محتوا قبل از دانلود

```
HEAD /seasonal/index-fall.html HTTP/1.1
Host: www.joes-hardware.com
Accept: *
```

پاسخ سرور (بدون Body):
```
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 617
```

### PUT
داده‌های ارسال‌شده از Client را روی سرور ذخیره می‌کند. محتوای Body را در Resource مشخص‌شده در URL می‌نویسد. 

> **نکته امنیتی که کتاب تأکید می‌کند:** چون PUT به کاربر اجازه تغییر محتوای سرور را می‌دهد، بسیاری از Web Server ها آن را Disable می‌کنند یا نیاز به احراز هویت دارند. 

### POST
برای ارسال داده به یک Gateway Application (مثل یک Form HTML یا یک API Endpoint) استفاده می‌شود. داده در Body قرار می‌گیرد. 

```
POST /inventory-check.cgi HTTP/1.1
Host: www.joes-hardware.com
Content-Type: application/x-www-form-urlencoded
Content-length: 27

product=1&quantity=5&price=150
```

### TRACE
یک Request تشخیصی است. درخواست را از طریق تمام Proxy ها و Gateway های مسیر ردیابی می‌کند. سرور نهایی تمام Message دریافت‌شده را **عیناً در Body Response** برمی‌گرداند — به این ترتیب Client می‌تواند ببیند پیامش در مسیر چطور تغییر کرده. 

```
TRACE /product-list.txt HTTP/1.1
Host: www.joes-hardware.com
Accept: text/*
```

پاسخ سرور:
```
HTTP/1.1 200 OK
Content-Type: message/http
Content-length: 96

TRACE /product-list.txt HTTP/1.1
Host: www.joes-hardware.com
Accept: text/*
```

> **محدودیت مهم:** TRACE نباید Body داشته باشد. 

### OPTIONS
می‌پرسد سرور برای یک Resource خاص از چه Method هایی پشتیبانی می‌کند. 

```
OPTIONS * HTTP/1.1
Host: www.joes-hardware.com
Accept: *
```

پاسخ سرور:
```
HTTP/1.1 200 OK
Allow: GET, POST, PUT, OPTIONS
```

### DELETE
از سرور می‌خواهد یک Resource را حذف کند. اما نویسندگان یک نکته مهم را ذکر می‌کنند: **سرور مجاز است این درخواست را نادیده بگیرد.** Client نمی‌تواند مطمئن باشد که حذف واقعاً انجام شده.

# فصل سوم — بخش دوم: Status Codes و Headers

## Status Codes — دسته‌بندی کامل

Status Code ها در پنج دسته کلی تقسیم می‌شوند. رقم اول کد، دسته را مشخص می‌کند: 

| دسته | محدوده | معنا |
|---|---|---|
| **Informational** | 100-199 | درخواست دریافت شد، پردازش ادامه دارد |
| **Success** | 200-299 | درخواست با موفقیت دریافت، فهمیده و پذیرفته شد |
| **Redirection** | 300-399 | برای تکمیل درخواست، اقدام بیشتری لازم است |
| **Client Error** | 400-499 | درخواست دارای خطای Syntax یا قابل اجرا نیست |
| **Server Error** | 500-599 | سرور در اجرای یک درخواست معتبر موفق نشد |

## دسته 100 — Informational

### 100 Continue
یک Status Code چالش‌برانگیز است. وقتی Client می‌خواهد یک Request با Body بزرگ ارسال کند، ابتدا یک درخواست اولیه با Header `Expect: 100-continue` می‌فرستد تا مطمئن شود سرور آماده دریافت است. 

اگر سرور آماده باشد، `100 Continue` می‌فرستد و Client Body را ارسال می‌کند. اگر سرور آماده نباشد، یک Error Code مناسب می‌فرستد و Client از ارسال Body جلوگیری می‌کند — این باعث صرفه‌جویی در Bandwidth می‌شود.

## دسته 200 — Success

### 200 OK
رایج‌ترین Response. درخواست موفق بوده. Body شامل Resource درخواست‌شده است. 

### 201 Created
برای درخواست‌های PUT یا POST که یک Resource جدید ایجاد کردند. سرور باید Location Header را با URL Resource جدید ارسال کند. 

### 202 Accepted
درخواست پذیرفته شده اما پردازش هنوز کامل نشده. سرور تضمینی نمی‌دهد که پردازش حتماً کامل شود. 

### 204 No Content
Response دارای Status Line و Headers است اما **Body ندارد.** اغلب بعد از یک عملیات موفق که نیازی به برگرداندن داده نیست استفاده می‌شود. 

### 206 Partial Content
بخشی از یک Resource برگردانده شده. این Status Code مربوط به **Range Request** هاست — وقتی Client فقط بخشی از یک فایل بزرگ را درخواست می‌کند. Header های `Content-Range` و `Date` الزامی هستند. 

## دسته 300 — Redirection

### 301 Moved Permanently
Resource به آدرس جدیدی منتقل شده. سرور آدرس جدید را در Header `Location` برمی‌گرداند. Client و هر Proxy باید URL های ذخیره‌شده را به‌روز کنند. 

### 302 Found
مشابه 301 اما **موقتی** است. Resource فعلاً در آدرس دیگری است اما در آینده به آدرس اصلی برمی‌گردد. Client نباید URL ذخیره‌شده را تغییر دهد. 

### 303 See Other
به Client می‌گوید Resource را از URL دیگری با Method `GET` بگیرد. معمولاً بعد از یک POST استفاده می‌شود تا از مشکل **Double Submit** (زدن دوباره دکمه Submit) جلوگیری شود. 

### 304 Not Modified
وقتی Client یک **Conditional Request** می‌فرستد (با Header هایی مثل `If-Modified-Since`)، اگر Resource تغییر نکرده باشد، سرور `304` برمی‌گرداند بدون Body. Client از نسخه Cache‌شده خودش استفاده می‌کند. 

### 307 Temporary Redirect
مشابه 302 است اما با یک تفاوت مهم: **Method و Body درخواست اصلی تغییر نمی‌کنند.** در 302، بعضی Client ها POST را به GET تبدیل می‌کردند که رفتار اشتباهی بود. 

## دسته 400 — Client Error

### 400 Bad Request
Request دارای Syntax اشتباه است یا سرور نمی‌تواند آن را بفهمد. 

### 401 Unauthorized
درخواست به احراز هویت نیاز دارد. سرور باید Header `WWW-Authenticate` را با نوع احراز هویت موردنیاز ارسال کند. 

### 403 Forbidden
سرور درخواست را فهمیده اما از اجرای آن امتناع می‌کند — **بدون توضیح دلیل.** برخلاف 401، احراز هویت کمکی نمی‌کند. 

### 404 Not Found
سرور Resource درخواست‌شده را پیدا نکرد. رایج‌ترین Error Code ای است که کاربران با آن روبه‌رو می‌شوند. 

### 405 Method Not Allowed
Method مورداستفاده برای این Resource مجاز نیست. سرور باید در Header `Allow` لیست Method های مجاز را برگرداند. 

### 408 Request Timeout
Client در زمان مشخص‌شده Request را کامل نکرد. 

### 409 Conflict
Request با وضعیت فعلی Resource در تضاد است — مثلاً آپلود یک فایل قدیمی‌تر از نسخه روی سرور. 

### 410 Gone
مشابه 404 اما با یک تفاوت: سرور **قطعاً می‌داند** که Resource وجود داشته و **برای همیشه** حذف شده. 

### 415 Unsupported Media Type
سرور نوع داده‌ای که Client فرستاده را پشتیبانی نمی‌کند. 

## دسته 500 — Server Error

### 500 Internal Server Error
عمومی‌ترین Server Error. سرور با یک خطای غیرمنتظره مواجه شده. 

### 501 Not Implemented
Client از یک قابلیت یا Method استفاده کرده که سرور از آن پشتیبانی نمی‌کند. 

### 502 Bad Gateway
یک Proxy یا Gateway پاسخ نامعتبری از سرور بعدی در زنجیره دریافت کرده. 

### 503 Service Unavailable
سرور موقتاً قادر به پاسخگویی نیست — معمولاً به دلیل حجم بالا یا در حال نگهداری (Maintenance). اگر مشخص باشد چه مدت طول می‌کشد، سرور باید Header `Retry-After` ارسال کند. 

### 504 Gateway Timeout
مشابه 408 اما برای Proxy یا Gateway — سرور بعدی در زنجیره در زمان مقرر پاسخ نداد. 

## Headers — جزئیات کامل

### General Headers
در هر دو Request و Response استفاده می‌شوند:

| Header | توضیح |
|---|---|
| `Connection` | نحوه مدیریت Connection بعد از Transaction |
| `Date` | تاریخ و زمان ساخت Message |
| `MIME-Version` | نسخه MIME استفاده‌شده |
| `Trailer` | مجموعه Header هایی که در Trailer یک Chunked Transfer هستند |
| `Transfer-Encoding` | نوع Encoding که روی Body اعمال شده |
| `Via` | Proxy هایی که Message از آنها عبور کرده |
| `Cache-Control` | دستورالعمل‌های Caching |

### Request Headers

| Header | توضیح |
|---|---|
| `Client-IP` | IP Address کاربر |
| `From` | آدرس ایمیل کاربر |
| `Host` | Hostname و Port سرور مقصد |
| `Referer` | URL صفحه‌ای که لینک فعلی از آن آمده |
| `UA-CPU` | نوع CPU کاربر |
| `User-Agent` | اطلاعات Browser یا Client |
| `Accept` | نوع‌های محتوایی که Client می‌پذیرد |
| `Accept-Charset` | Character Set هایی که Client می‌پذیرد |
| `Accept-Encoding` | Encoding هایی که Client می‌پذیرد |
| `Accept-Language` | زبان‌هایی که Client ترجیح می‌دهد |
| `Authorization` | داده احراز هویت |
| `Cookie` | Token هایی که سرور قبلاً ارسال کرده |
| `If-Modified-Since` | فقط اگر بعد از این تاریخ تغییر کرده برگردان |
| `If-None-Match` | فقط اگر ETag تغییر کرده برگردان |

### Response Headers

| Header | توضیح |
|---|---|
| `Age` | چه مدت است که Response در Cache بوده |
| `Server` | نوع و نسخه نرم‌افزار سرور |
| `Allow` | Method های مجاز برای این Resource |
| `Location` | URL Redirect برای Redirection |
| `Set-Cookie` | Token ای که سرور می‌خواهد Client ذخیره کند |
| `WWW-Authenticate` | نوع احراز هویت موردنیاز |

### Entity Headers

| Header | توضیح |
|---|---|
| `Content-Type` | نوع MIME محتوای Body |
| `Content-Length` | طول Body به Byte |
| `Content-Encoding` | Encoding اعمال‌شده روی Body |
| `Content-Language` | زبان محتوای Body |
| `Content-Location` | آدرس جایگزین برای Resource |
| `Content-MD5` | Checksum MD5 از محتوا برای تأیید صحت |
| `Content-Range` | محدوده Byte های ارسال‌شده در یک Partial Response |
| `Last-Modified` | آخرین تاریخ تغییر Resource |
| `Expires` | تاریخ انقضای محتوا |
| `ETag` | یک شناسه یکتا برای نسخه فعلی Resource |

# فصل چهارم — بخش اول: TCP Connections و Performance

## چرا Connection Management مهم است؟

نویسندگان این فصل را با یک ادعای صریح شروع می‌کنند: **اکثر توسعه‌دهندگان HTTP درک درستی از TCP Connections ندارند.** این سوءتفاهم مستقیماً روی Performance اپلیکیشن تأثیر می‌گذارد. 

HTTP پیام‌هایش را از طریق **TCP Connection** منتقل می‌کند. قبل از اینکه بتوان یک HTTP Request فرستاد، باید یک TCP Connection بین Client و Server برقرار شود.

## TCP Connection — زیرساخت انتقال HTTP

TCP داده را به صورت **جریان پیوسته‌ای از Byte ها** منتقل می‌کند. این Byte ها در قطعاتی به نام **Segment** بسته‌بندی می‌شوند و از طریق IP منتقل می‌شوند. هر Segment در یک **IP Packet** قرار می‌گیرد. 

هر IP Packet شامل این اجزاست: 

- **IP Packet Header:** آدرس IP مبدأ و مقصد، اندازه و سایر Flag ها
- **TCP Segment Header:** شماره Port مبدأ و مقصد، Flag های کنترلی و اطلاعات ترتیب داده
- **TCP Data Chunk:** بخشی از داده واقعی

## TCP Performance — کجا زمان تلف می‌شود؟

کتاب دقیقاً مشخص می‌کند که در یک HTTP Transaction زمان صرف چه کارهایی می‌شود: 

```
DNS Lookup → TCP Connection → HTTP Request → Processing → HTTP Response
     │               │               │             │              │
  ~ms تا s       Three-Way       Header+Body    Server Side   Header+Body
               Handshake          ارسال          پردازش        دریافت
```

مهم‌ترین تأخیرها:

### 1. TCP Connection Setup — Three-Way Handshake

قبل از اینکه یک Byte از داده HTTP منتقل شود، باید TCP Handshake انجام شود: 

```
Client                          Server
  │                               │
  │──── SYN ──────────────────────▶│   (۱) Client می‌گوید: «می‌خواهم وصل شوم»
  │                               │
  │◀─── SYN + ACK ────────────────│   (۲) Server می‌گوید: «قبول دارم»
  │                               │
  │──── ACK + HTTP Request ───────▶│   (۳) Client تأیید می‌کند + Request را می‌فرستد
  │                               │
```

این Handshake **یک Round-Trip کامل** قبل از هر Request اضافه می‌کند. برای Connection های طولانی این هزینه یک‌بار پرداخت می‌شود، اما برای Connection های کوتاه هر بار باید تکرار شود. 

### 2. TCP Slow Start — آغاز کند اتصال

TCP برای جلوگیری از ازدحام شبکه (Congestion) از مکانیزمی به نام **Slow Start** استفاده می‌کند. یک Connection جدید نمی‌تواند فوراً با حداکثر سرعت داده بفرستد — باید ابتدا با سرعت کم شروع کند و به تدریج سرعت را افزایش دهد. 

نتیجه عملی: **Connection های قدیمی از Connection های جدید سریع‌تر هستند.** یک Connection که مدتی در حال انتقال داده بوده، Slow Start را پشت سر گذاشته و می‌تواند با سرعت کامل کار کند.

> این یکی از دلایل اصلی است که **Persistent Connection** ها در HTTP/1.1 اضافه شدند.

### 3. Nagle Algorithm و TCP_NODELAY

TCP یک الگوریتم دارد به نام **Nagle Algorithm** که Packet های کوچک را با هم ترکیب می‌کند تا تعداد Packet های شبکه را کاهش دهد. 

اما این الگوریتم برای HTTP مشکل‌ساز است چون:
- HTTP Header ها معمولاً کوچک هستند
- Nagle Algorithm آنها را نگه می‌دارد تا با داده بیشتری ترکیب کند
- این باعث تأخیر می‌شود

راه‌حل: فعال کردن `TCP_NODELAY` که Nagle Algorithm را غیرفعال می‌کند. 

### 4. TIME_WAIT و Port Exhaustion

وقتی یک TCP Connection بسته می‌شود، آدرس IP و شماره Port به مدت **2MSL (حدود 2 دقیقه)** در حالت `TIME_WAIT` باقی می‌مانند. 

این برای جلوگیری از تداخل بین Connection های قدیمی و جدید است. اما در سرورهای پرترافیک می‌تواند منجر به **Port Exhaustion** شود — کمبود Port در دسترس برای Connection های جدید.

## HTTP Connection Handling — مدیریت Connection ها

### Connection Header

HTTP یک Header به نام `Connection` دارد که می‌تواند سه نوع Token داشته باشد: 

- **نام یک HTTP Header:** آن Header فقط برای این Connection معتبر است و نباید به Downstream فرستاده شود
- **`close`:** بعد از این Transaction، Connection بسته می‌شود
- **`keep-alive`:** می‌خواهیم Connection را باز نگه داریم (HTTP/1.0)

> **نکته مهم:** هر Proxy که `Connection` Header را دریافت می‌کند باید آن را **پردازش و حذف** کند قبل از اینکه Message را به جلو بفرستد. 

## انواع Connection — چهار رویکرد

### 1. Sequential Connections — ساده‌ترین، کندترین

```
Connection 1:  [Connect][Request][Response][Close]
Connection 2:                                      [Connect][Request][Response][Close]
Connection 3:                                                                          [Connect]...
```

برای هر درخواست یک Connection جدید باز می‌شود. کند است چون هر بار باید Handshake و Slow Start انجام شود. 

### 2. Parallel Connections — چند Connection همزمان

چند HTTP Transaction به‌صورت همزمان روی Connection های مختلف انجام می‌شوند. 

```
Connection 1: [Connect][──── Request 1 ────]
Connection 2: [Connect][──── Request 2 ────]
Connection 3: [Connect][──── Request 3 ────]
Connection 4: [Connect][──── Request 4 ────]
```

**مزایا:**
- زمان کل کاهش می‌یابد
- اگر یک Connection کند باشد، بقیه ادامه می‌دهند

**معایب:**
- هر Connection به Memory و منابع نیاز دارد
- اگر Bandwidth محدود باشد، Parallel Connection ها با هم رقابت می‌کنند و ممکن است نتیجه بدتری بدهند
- سرور باید بار Connection های بیشتری را تحمل کند 

> **محدودیت عملی:** مرورگرها معمولاً تعداد Parallel Connection ها را به **6-8** محدود می‌کنند.

### 3. Persistent Connections — مهم‌ترین بهبود HTTP/1.1

بعد از انجام یک Transaction، Connection **باز می‌ماند** و برای Request های بعدی استفاده می‌شود. 

```
Connection:  [Connect][Request 1][Response 1][Request 2][Response 2][Request 3][Response 3][Close]
```

**مزایا:**
- هزینه Handshake فقط یک‌بار پرداخت می‌شود
- TCP Slow Start اثر کمتری دارد
- تعداد Connection های باز کمتر است

**دو نوع Persistent Connection:**

**Keep-Alive (HTTP/1.0):** باید صریحاً با Header `Connection: keep-alive` درخواست شود. سرور می‌تواند این پارامترها را تنظیم کند: 

```
Connection: keep-alive
Keep-Alive: max=5, timeout=120
```

- `max=5`: حداکثر ۵ Transaction دیگر روی این Connection
- `timeout=120`: اگر ۱۲۰ ثانیه بی‌استفاده باشد بسته شود

**Persistent (HTTP/1.1):** در HTTP/1.1 تمام Connection ها به‌صورت **پیش‌فرض Persistent** هستند. برای بستن Connection باید صریحاً `Connection: close` فرستاده شود. 

### 4. Pipelined Connections — ارسال چند Request بدون انتظار

**Pipelining** اجازه می‌دهد چند Request پشت سر هم فرستاده شوند **بدون اینکه منتظر Response بمانیم.** 

```
Client: [Request 1][Request 2][Request 3]───────▶
Server:                        ◀─────────────────[Response 1][Response 2][Response 3]
```

**محدودیت‌های مهم:**
- فقط روی Persistent Connection ها کار می‌کند
- Response ها باید دقیقاً **به همان ترتیب** Request ها برگردند
- اگر سرور Connection را ببندد، Client باید Request های بی‌پاسخ را دوباره بفرستد
- فقط برای **Idempotent Method** ها مناسب است (مثل GET و HEAD، نه POST)

# فصل چهارم — بخش دوم: Connection Close و رفتارهای پیچیده TCP

## بستن Connection — ساده به نظر می‌رسد، اما نیست

نویسندگان این بخش را **«رازهای Connection Close»** می‌نامند.  دلیلش این است که بستن یک TCP Connection ظاهراً ساده است، اما جزئیاتی دارد که می‌توانند منجر به از دست رفتن داده شوند. 

## At Will Close — بستن دلخواه Connection

هر دو طرف Client و Server می‌توانند در هر زمانی Connection را ببندند — حتی در وسط یک Transaction. اما کتاب تأکید می‌کند که این کار باید با احتیاط انجام شود. 

سرورها معمولاً Connection را در این موارد می‌بندند:
- بعد از ارسال آخرین Response
- وقتی Connection برای مدت طولانی بی‌استفاده بوده (Idle Timeout)
- وقتی تعداد Transaction های مجاز (`max` در Keep-Alive) تمام شده

## دو نوع TCP Close

TCP دو مکانیزم برای بستن Connection دارد که تفاوت مهمی بین آنهاست: 

### Full Close
هر دو کانال ارسال و دریافت بسته می‌شوند.

```
Client ──────────────────── Server
  │   ◀── FIN ──────────────  │   (Server کانال ارسالش را می‌بندد)
  │   ──── ACK ──────────────▶│
  │   ──── FIN ──────────────▶│   (Client کانال ارسالش را می‌بندد)
  │   ◀── ACK ───────────────  │
```

### Half Close
فقط یک کانال بسته می‌شود — معمولاً کانال **ارسال (Output)**. 

```
Client ──── Half Close Output ───▶ Server
  │         (من دیگر نمی‌فرستم)        │
  │   ◀─── داده هنوز می‌تواند بیاد ────│
```

Half Close در HTTP اهمیت دارد چون به یک طرف اجازه می‌دهد بگوید «من کارم تمام شد» اما هنوز آماده دریافت داده باشد.

## Reset Error — خطرناک‌ترین رفتار

اگر یک طرف داده‌ای به یک Connection **بسته‌شده** بفرستد، طرف مقابل یک **TCP Reset** برمی‌گرداند. 

مشکل اینجاست: **TCP Reset می‌تواند Buffer های دریافت را پاک کند.** این یعنی اگر سرور یک Full Close انجام داده باشد و Client هنوز داده می‌فرستد، وقتی Reset دریافت می‌شود، ممکن است داده‌ای که سرور قبلاً فرستاده و هنوز در Buffer است از بین برود — حتی اگر آن داده کاملاً معتبر بوده باشد. 

## Graceful Connection Close — روش صحیح بستن

برای جلوگیری از Reset Error، کتاب یک رویه مشخص توصیه می‌کند: 

**مراحل Graceful Close:**

1. **Half Close کردن Output:** فقط کانال ارسال را ببند
2. **منتظر ماندن:** صبر کن تا طرف مقابل هم Output خودش را ببندد، یا یک Timeout رخ دهد
3. **Full Close کردن:** حالا Connection را کاملاً ببند

```
طرفی که می‌خواهد ببندد:
  │── Half Close Output ──▶   (من دیگر داده نمی‌فرستم)
  │◀── داده‌های باقی‌مانده ──   (هنوز می‌خوانم)
  │◀── Half Close Output ──   (طرف مقابل هم تمام کرد)
  │── Full Close ──────────▶  (حالا می‌توان کاملاً بست)
```

> **نکته عملی:** در .NET Core این رفتار از طریق `Socket.Shutdown(SocketShutdown.Send)` قبل از `Socket.Close()` پیاده‌سازی می‌شود.

## جمع‌بندی Performance — راهکارهای عملی

کتاب در پایان فصل یک جمع‌بندی از راهکارهای بهبود Performance ارائه می‌دهد: 

### Parallel Connections
برای صفحاتی با منابع زیاد (تصویر، CSS، JS) مناسب است. اما تعداد زیاد Connection همزمان می‌تواند روی سرور فشار بیاورد.

**مناسب برای:** بارگذاری اولیه صفحات وب

### Persistent Connections
بهترین راه‌حل برای اکثر سناریوها. هزینه Handshake و Slow Start را برای چندین Request جبران می‌کند.

**مناسب برای:** API Call های متعدد به یک سرور (مثلاً در یک Service مبتنی بر .NET Core که به چند Endpoint مراجعه می‌کند)

### Persistent + Parallel ترکیب
در عمل مرورگرهای مدرن از ترکیب هر دو استفاده می‌کنند: چند Persistent Connection موازی. 

### Pipelined Connections
کمترین Latency را دارد اما پیچیده‌ترین پیاده‌سازی را هم نیاز دارد. فقط برای Method های Idempotent مناسب است.

## یک دیدگاه انتقادی

نویسندگان در پایان فصل صادقانه اعتراف می‌کنند که **مستندات HTTP/1.1 درباره Connection Management ناقص و گاهی متناقض هستند.**  بسیاری از رفتارهای واقعی در دنیای عملی از طریق آزمون و خطا کشف شده‌اند، نه از طریق مستندات رسمی. 

این توضیح می‌دهد که چرا پیاده‌سازی‌های مختلف (Browser ها، Proxy ها، Server ها) گاهی رفتارهای متفاوتی دارند — همه آنها استاندارد را **به شکل خود** تفسیر کرده‌اند.
